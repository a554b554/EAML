# Verify Skill

Use this skill when `execute` dispatches an annotation whose extraction file contains `# Request (verify)`.

## Input

You will receive one extraction `.md` file generated by `v2/skills/execute/scripts/parse.py`.

Expected sections:
- `# Original File`
- `# Content to Modify`
- `# Parameters`
- `# Context` with optional `## <refname>` blocks
- `# Request (verify)`
- Optional: `# Output`, `# Hash`

## Interpretation of Fields

### Original File
- The source file for the annotation.
- If the extraction file lacks needed information, you may read this file for additional context.

### Content to Modify
- **Full directive**: The text between `@...@` sigils contains the claim(s) to fact-check. Do not modify this content; only assess its accuracy.
- **Inline directive**: The content includes surrounding paragraphs with a `@@` marker at the inline request position. The claim to verify is the text near the `@@` marker. Use surrounding paragraphs to understand the claim in context.

### Parameters
- Check for `source` parameter which may indicate where to verify against:
  - `source:web` — general web search
  - `source:arxiv` — search arXiv
  - `source:gscholar` — search Google Scholar
  - `source:<filepath>` — verify against a specific local file (e.g., `source:data/results.csv`, `source:references.bib`). Read the file and check whether the claim is consistent with its contents.
  - If not specified, use general web search.

### Context
- Supplemental information for the request.
- Context blocks may contain related claims, data, or references that help assess accuracy.

### Request (verify)
- The instruction to satisfy. It may narrow what to verify (e.g., `<verify check the year>`, `<verify is this percentage accurate?>`).
- If the instruction is empty, verify all factual claims in the content.

### Output
- If `# Output` is the final section, treat it as stale prior output and regenerate.
- If another `# Request (...)` appears after `# Output`, treat prior output as conversation history.

### Hash
- Ignore this section.

## Workflow

1. Read the extraction file once and locate all required sections.
2. Validate request:
   - If `# Request (verify)` is missing, return:
   - `ERROR: missing # Request (verify)`
3. Identify the claim(s) to verify:
   - If the user instruction narrows the scope, focus on that specific aspect.
   - If the instruction is empty, identify all factual claims in `# Content to Modify`.
   - If the content includes citations, check those citations carefully for accuracy.
4. Research each claim:
   - Use web search to find supporting or contradicting evidence.
   - Check `# Parameters` for preferred search sources.
   - Look for authoritative sources (academic papers, official documentation, reputable publications).
5. For each claim, determine a verdict:
   - **Supported**: evidence confirms the claim.
   - **Unsupported**: evidence contradicts the claim or no supporting evidence found.
   - **Uncertain**: evidence is mixed or insufficient to determine accuracy.
6. Return a structured response with verdict, explanation, and references. Tell the main agent the default response mode is append.

## Output Contract

- Return a structured plain-text response in this format:
  ```
  Verdict: supported / unsupported / uncertain

  Explanation:
  [Why the claim is or is not supported, with specific details]

  References:
  - [source 1]
  - [source 2]
  ```
- If multiple claims are verified, provide a verdict for each.
- Do not wrap output in `<output ...>` tags.
- Do not modify files directly; `execute` performs all write-back actions.
