# Plan Skill

Use this skill when `execute` dispatches an annotation whose extraction file contains `# Request (plan)`.

## Input

You will receive one extraction `.md` file generated by `v2/skills/execute/scripts/parse.py`.

Expected sections:
- `# Original File`
- `# Content to Modify`
- `# Parameters`
- `# Context` with optional `## <refname>` blocks
- `# Request (plan)`
- Optional: `# Output`, `# Hash`

## Interpretation of Fields

### Original File
- The source file for the annotation.
- Read this file to understand the full document structure. The plan skill typically needs the complete document to produce meaningful revision suggestions.

### Content to Modify
- **Full directive**: The text between `@...@` sigils is the scope of the revision plan. Focus suggestions on this content, but consider the broader document for coherence.
- **Inline directive**: The content includes surrounding paragraphs with a `@@` marker. The revision plan should focus on the area around `@@`, using surrounding paragraphs for context. Consider reading `# Original File` for the full picture.

### Parameters
- Ignore this section unless the user request explicitly depends on a parameter value.

### Context
- Supplemental information for the request.
- Context blocks provide additional material that may inform the revision plan (e.g., style guides, reference sections, related content).

### Request (plan)
- The revision goal to plan for (e.g., `make the tone more formal`, `restructure for clarity`, `add more technical detail`).
- Treat it as the highest-priority directive.

### Output
- If `# Output` is the final section, treat it as stale prior output and regenerate.
- If another `# Request (...)` appears after `# Output`, treat prior output as conversation history.

### Hash
- Ignore this section.

## Workflow

1. Read the extraction file once and locate all required sections.
2. Validate request:
   - If `# Request (plan)` is missing, return:
   - `ERROR: missing # Request (plan)`
3. Analyze the content:
   - Read `# Content to Modify` to understand the current state.
   - Read `# Original File` if the full document context is needed.
   - Review `# Context` blocks for supporting information.
4. Produce a revision plan that:
   - Addresses the revision goal stated in `# Request (plan)`
   - Lists specific, actionable changes
   - Explains the reasoning behind each suggested change
   - Preserves protected text (`<<...>>`) — do not suggest modifying protected content
   - Considers the document's overall coherence and structure
5. Return the structured plan.

## Output Contract

- Return a structured plain-text revision plan:
  ```
  Revision goal: [restate the goal]

  Suggested changes:
  1. [specific change] — [reason]
  2. [specific change] — [reason]
  ...

  Notes:
  [any overall observations or dependencies between changes]
  ```
- Do not modify the document content — only suggest changes.
- Do not wrap output in `<output ...>` tags.
- Do not modify files directly; `execute` performs all write-back actions.
